# 后端开发面试标准答案（真题实战版）

## 一、计算机基础 (网络、计组与系统)

### Q5: 进程与线程的区别？协程的角色？
**A:** 
- **进程**：资源分配的最小单位，拥有独立的地址空间。
- **线程**：执行调度的最小单位，同进程内多线程共享内存（堆和方法区），但有私有的栈和程序计数器。
- **协程**：用户态轻量级线程，由程序员调度。在 Go、Kotlin 或 Python (asyncio) 中常用。*作用*：极大地减少了线程上下文切换带来的开销，适合高并发 I/O 绑定场景。

## 二、Java 核心与并发编程

### Q9: 如何保证堆中分配内存不产生冲突？什么是 TLAB？
**A:** **TLAB (Thread Local Allocation Buffer)**：在 Java 堆的伊甸园区 (Eden) 为每个线程分配一块私有的内存缓冲区。
- **核心逻辑**：当线程需要分配对象内存时，首先在自己的 TLAB 中查找空间。这避免了多线程在同一个 Eden 区域争夺内存指针（竞争 CAS）。
- **同步**：只有当 TLAB 满并申请新的 TLAB 时，才需要使用 CAS 进行线程同步。

### Q10: 什么是死锁（Deadlock）？
**A:** 四个必要条件：1. 互斥条件。2. 请求与保持。3. 不可剥夺。4. 循环等待。
- **排查**：使用 `jstack` 查看线程堆栈，识别 "Found one Java-level deadlock"。
- **预防**：按顺序加锁、使用 `tryLock` 定时尝试、减少持有锁的时间。

## 三、Spring 框架与架构设计

### Q15: Spring 循环依赖与注解问题？
**A:** 三级缓存只能解决 **Setter 注入** 的单例循环依赖。
- **异常场景**：1. **构造器注入**（无法通过缓存暴露未初始化对象）。2. **使用 @Async 或 @Transaction 的早期代理**。如果某 Bean 在初始化后由于 AOP 产生了代理对象，而缓存中存的是原始对象，则会报错。*解决*：使用 `@Lazy` 或手动调整依赖关系。

## 四、存储引擎与中间件 (MySQL, Redis, MQ)

### Q17: MySQL 为什么选 B+ 树？相比 LSM Tree？
**A:** 
- **B+ 树 (InnoDB)**：随机读性能极佳，层级低。叶子节点有序链表，便于范围查询。数据全在叶子，内部节点仅索引，提高缓存命中率。
- **LSM Tree (LevelDB/HBase)**：顺序写性能极好（追加写），但读性能较弱（需多层合并）。适合日志系统、监控等写多读少场景。

### Q21: 分布式锁与幂等性 QPS 优化？
**A:** 分布式锁确实会显著降低 QPS。
- **优化设计**：1. **分段锁**（如果业务允许）。2. **本地内存二级缓存**（预先过滤重复请求）。3. **基于数据库唯一键约束 (Unique Key)**：将幂等性逻辑下移到 DB 层，由 DB 保证原子性。4. **乐观锁 (Version 版本号)**：减少悲观锁的阻塞。

## 五、AI 与 RAG 专项架构 (LlamaIndex vs LangChain)

### Q24 & Q27: RAG 架构与评测策略
**A:** 
- **离线层**：文档加载 -> 语义切分 (RecursiveCharacterTextSplitter) -> Embedding 生成 (OpenAI-v3 / BGE) -> VectorStore。
- **评测手段**：RAGAS 模型（忠实度、相关度评测）、数字化的 Hit Rate (命中率)、MRR (平均倒数排名)。

### Q31: 多模型降级（LlamaIndex vs LangChain 对比）
**A:** 
- **LangChain**：擅长灵活的 **Chain (链式调用)**，通过 `RunnableSerializable` 实现 Fallback。偏重“胶水代码”，生态极其丰富，但过于笨重。
- **LlamaIndex**：深度专注于 **Data Framework (数据框架)**。其 `RouterQueryEngine` 内置了更高效的索引分流和索引选择逻辑。在检索端（Retriever）的优化比 LangChain 更深，提供了更多高级分块与重排序算子。
- **降级感知**：自动降级通常由接口超时（Timeout）或 429 (Rate Limit) 触发。状态机：Normal -> Degraded (尝试备用模型) -> Failsafe (抛出异常/返回静态答案)。

### Q32: MCP 与 Function Calling 的区别？
**A:** 
- **Function Calling**：是模型原生能力，输出格式化的 JSON 供应用调用。
- **MCP (Model Context Protocol)**：是由 Anthropic 提出的标准化协议。它允许模型直接与一套标准化的“服务器/工具集”交互。
- **实现方案**：Function Calling 需要开发者手动解析参数并写 RPC/HTTP 调用；MCP 提供了插件式的架构，只要符合协议，模型可以无缝调用各种工具集。

## 六、性能调优与场景实战 (Fermi & Systematic)

### Q37: [费米估算] 学校每天外卖订单量
**A:** 
1. **基数估算**：学校学生 30,000 人。
2. **频率估算**：平均每人每天点外卖概率 0.3（有些人从不点，有些人点两次）。30,000 * 0.3 = 9,000 单。
3. **节假日修正**：周末和考试周的波动率约 1.2x。结论：约 1 万单/日。
4. **后端优化建议**：1. **GEO 索引优化**（提升商家匹配速度）。2. **个性化推荐召回**（提升转化率即订单量）。3. **写避让策略**（高并发秒杀期保护下单主链路）。

### Q38: 如何识别面试 AI 辅助作弊？
**A:** **系统化方案**：
1. **前端层**：检测页面焦点切换 (Visibility API)、禁止 F12/右键、捕获剪切板粘贴行为。
2. **数据监控**：对候选人打字节律进行分析（AI 生成的文本往往是“瞬间涌入”或节律机械）。
3. **AI 逆向检测**：使用专门的 AI Detector 模型判断文本生成特征。
4. **题库动态化**：加大开放性题目比例，避免标准化“八股文”答案被辅助工具轻易秒杀。
